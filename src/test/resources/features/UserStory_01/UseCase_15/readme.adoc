[#UC_01_15]
== UseCase_01_15: Client ressource anfrage fachdienst SC 400

.UseCase_01_15
[cols="1,1a",options="autowidth"]
|===
| *Titel:* | Client ressource anfrage fachdienst SC 400

| Akteur: |
*MA einer LEI*

| weitere Akteure: |
- *ZETA Client*
- *PEP HTTP Proxy*

| Auslöser: |
Ein MA einer LEI fordert eine Ressource über den ZETA Client an.

| Kurzbeschreibung: |
Der ZETA Client sendet eine Anfrage mit einem Access Token für eine geschützte Ressource an den PEP HTTP Proxy.
Der DPoP Header fehlt, obwohl er nach Richtlinie zwingend erforderlich ist.
Der PEP HTTP Proxy beantwortet die Anfrage mit ``400 Bad Request``.

| Vorbedingung: |
Der ZETA Client ist registriert, authentisiert und autorisiert.
Die Resource-Anfrage erfordert einen DPoP Nachweis (``DPoP``-Header).

| Nebenbedingung: |
Der DPoP Header wird beim Versand der Anfrage unterdrückt oder vom Client nicht übermittelt.

| Ablauf: |
. MA startet den Abruf einer geschützten Ressource über den ZETA Client.
. Der ZETA Client erzeugt oder lädt das Access Token sowie den DPoP Nachweis.
. Der ZETA Client ruft die Ressource am PEP HTTP Proxy auf:
- ``GET /resource``
- ``Authorization: DPoP ... (Access Token)``
- **kein** ``DPoP``-Header liegt bei der Anfrage vor.
. Der PEP HTTP Proxy prüft die übermittelten Header und stellt fest, dass der DPoP Nachweis fehlt, obwohl er zwingend verlangt wird.
. Der PEP HTTP Proxy antwortet mit ``400 Bad Request`` und erläutert im Fehlertext die fehlende PoP-Information.
. Der ZETA Client informiert den MA, dass die Anfrage unvollständig war und der DPoP Nachweis ergänzt werden muss.

| Nachbedingung: |
Der Ressourcenzugriff wurde abgewiesen.
Der Ressourcenzugriff wurde abgewiesen. Der MA muss die Anfrage mit PoPP-Header erneut senden.
