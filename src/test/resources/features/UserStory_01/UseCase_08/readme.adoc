== UseCase_01_08: Client authentisierung und autorisierung software attest SC 401

.UseCase_01_08
[cols="1,1a",options="autowidth"]
|===
| *Titel:* | Client authentisierung und autorisierung software attest SC 401

| Akteur: |
*MA einer LEI*

| weitere Akteure: |
- *ZETA Client*
- *PDP Authorization Server*

| Auslöser: |
Ein MA einer LEI möchte auf einen Resource Server zugreifen und muss sich dafür authentisieren und autorisieren.

| Kurzbeschreibung: |
Um auf einen Resource Server zugreifen zu können, benötigt der
Client ein Access Token vom PDP Authorization Server.
Stationäre ZETA Clients verwenden dafür den Token Exchange Flow mittels Software-Attestierung.
Der erste Token-Request wird aufgrund eines ungültigen DPoP Proofs mit ``401 Unauthorized`` beantwortet.
Der ZETA Client befolgt die Anweisung zur erneuten Authentifizierung, führt den Token Exchange erneut aus und erhält anschließend gültige Access Tokens.

| Vorbedingung: |
Der stationäre ZETA Client ist beim ZETA Guard registriert und besitzt eine ``client_id`` samt Schlüsselpaar.
Es liegt kein gültiges Access Token vor.

| Nebenbedingung: |
Keine zusätzlichen Nebenbedingungen.

| Ablauf: |
1. MA startet den *Authentisierung-/Autorisierungs-Prozess* im ZETA Client.
2. Der ZETA Client fordert beim PDP Authorization Server eine Nonce an (``GET /nonce``).
3. Der PDP Authorization Server antwortet mit ``200 OK`` und Nonce.
4. Der ZETA Client erstellt Client Assertion und ungültigen (z.B. abgelaufenen)  DPoP Proof für den Token-Austausch (Software-Attestierung).
5. Der ZETA Client sendet den Token-Request an den PDP Authorization Server (``POST /token``).
6. Der PDP Authorization Server antwortet mit ``401 Unauthorized``.
7. Der ZETA Client fordert erneut eine Nonce an (``GET /nonce``).
8. Der PDP Authorization Server antwortet mit ``200 OK`` und Nonce.
9. Der ZETA Client erstellt Client Assertion und DPoP Proof erneut.
10. Der ZETA Client sendet den Token-Request erneut (``POST /token``).
11. Der PDP Authorization Server antwortet mit ``200 OK`` und Access bzw. Refresh Token.
12. Der ZETA Client informiert den MA über die erfolgreiche Zugriffrechtserteilung.

| Nachbedingung: |
Der stationäre ZETA Client kann auf einen Resource Server zugreifen, d.h. dieser ist authentisiert und autorisiert.
