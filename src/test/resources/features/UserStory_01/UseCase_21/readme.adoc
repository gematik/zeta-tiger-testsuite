[#UC_01_21]
== UseCase_01_21: Client ressource anfrage fachdienst waehrend aktualisierung SC 200

.UseCase_01_21
[cols="1,1a",options="autowidth"]
|===
| *Titel:* | Client ressource anfrage fachdienst waehrend aktualisierung SC 200

| Akteur: |
*MA einer LEI*

| weitere Akteure: |
- *ZETA Client*
- *PEP HTTP Proxy*
- *Resource Server*

| Auslöser: |
Ein MA einer LEI fordert eine Ressource über den ZETA Client an, während im Hintergrund ein Versionswechsel oder Rollback am Deployment durchgeführt wird.

| Kurzbeschreibung: |
Der ZETA Client sendet Anfragen an den Fachdienst über den PEP HTTP Proxy. +
Während eine langlaufende Anfrage verarbeitet wird, wird ein Versionswechsel des Deployments ausgelöst. +
Dabei wird nachgewiesen, dass die neue Version im Hintergrund heruntergeladen wird, während die Anfrage weiterhin offen ist. +
Die laufende Anfrage wird dennoch ordnungsgemäß mit ``200 OK`` abgeschlossen. +
Anschließend wird geprüft, dass weitere Anfragen bis zur finalisierten Aktualisierung ohne Verfügbarkeitsunterbrechung erfolgreich verarbeitet werden. +
Zusätzlich wird ein fehlerhaftes Update mit einem nicht verfügbaren Image ausgelöst, der fehlgeschlagene Rollout nachgewiesen und danach die automatische Rückkehr auf die stabile Vorversion innerhalb eines festen Zeitbudgets verifiziert.

| Vorbedingung: |
Die beteiligten Komponenten sind erreichbar und das Deployment des PEP HTTP Proxy kann aktualisiert bzw. zurückgerollt werden.

| Nebenbedingung: |
Ein Versionswechsel (inkl. möglichem Rollback) findet während laufender Anfragen statt.

| Ablauf: |
. MA startet über den ZETA Client eine langlaufende Anfrage an den Fachdienst.
. Der ZETA Client ruft den PEP HTTP Proxy auf.
. Während die Anfrage verarbeitet wird, wird am Deployment des PEP HTTP Proxy ein Versionswechsel ausgelöst.
. Dabei wird nachgewiesen, dass die neue Version im Hintergrund heruntergeladen wird, ohne die laufende Anfrage zu unterbrechen.
. Die bereits laufende Anfrage wird erfolgreich abgeschlossen und mit ``200 OK`` beantwortet.
. Direkt nach dem Versionswechsel werden weitere Anfragen gesendet.
. Diese Folgeanfragen werden ohne Verfügbarkeitsunterbrechung bis zum Abschluss des Rollouts ebenfalls mit ``200 OK`` beantwortet.
. Im Anschluss wird ein fehlerhaftes Update mit einem nicht verfügbaren Image ausgelöst.
. Der fehlgeschlagene Rollout wird nachgewiesen.
. Danach wird die automatische Rückkehr des Deployments auf die stabile Vorversion innerhalb eines festen Zeitbudgets geprüft.
. Nach der Rückkehr auf die stabile Vorversion wird erneut eine Anfrage gesendet.
. Die Anfrage nach der Rückkehr auf die stabile Vorversion wird erfolgreich mit ``200 OK`` beantwortet.

| Nachbedingung: |
Aktualisierungen werden im Hintergrund heruntergeladen, ohne laufende Anfragen zu beeinträchtigen. +
Laufende Anfragen werden trotz Versionswechsel erfolgreich abgeschlossen. +
Weitere Anfragen bleiben während des Übergangs bis zur finalisierten Aktualisierung verfügbar. +
Ein fehlerhaftes Update wird erkannt und nicht erfolgreich aktiviert. +
Nach einem Rollback steht wieder eine stabile Vorversion zur Verfügung.

    Note over PEP-HP: Rollback auf stabile Vorversion
    MA->>+ZC: Anfrage nach Rollback
    ZC->>+PEP-HP: GET /helloZeta
    PEP-HP->>+RS: Weiterleiten der Anfrage
